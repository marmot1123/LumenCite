## 0. プロダクト文脈

**LumenCite** はデスクトップの文献（参考文献）管理アプリ。Tauri 2 + React 18 + TypeScript。研究者が論文・書籍を管理し、PDF を読み、LLM に要約させる。

v0.2.0 で追加する **LLM Chat** は、LumenCite を「研究の壁打ち相手」にする機能。ユーザーが自然言語で質問すると、LLM が**ライブラリ内の全文検索（FTS5）をツール経由で反復しながら**、複数文献を横断して回答する。LLM はタグ付け・ノート追記などの **DB 書き換えツール**や、外部 **MCP サーバー**のツールも（承認制で）呼べる。

Chat は**独立スクリーン**として実装する（ライブラリ画面・詳細画面と並ぶ第3の画面）。

## 1. デザイントーン / 既存資産との整合

既存アプリのデザイン言語を踏襲すること。新しい視覚言語を作らない。

- **テーマ**: light / dark / **auto**（OS 追従）の3モード。すべての色は両テーマで成立させる
- **アクセントカラー**: `amber` / `indigo` / `teal` / `rose` の4種をユーザーが選択。主要 CTA・選択状態・リンクにアクセントを使う
- **行密度**: `compact` / `default` / `comfortable`。リスト系（セッション一覧・メッセージ）は密度設定に追従
- **多言語**: 日本語 / 英語。文字列は可変長前提（日↔英でラベル幅が変わる）
- **既存の詳細ビュー**: 3ペイン（左サムネ96px / 中央PDF / 右メタパネル340px）。Chat のサイドバー幅・余白はこの世界観に合わせる
- **本文レンダリング**: assistant メッセージは Markdown + 数式（KaTeX、`$…$` / `$$…$$`）。既存の `MathMarkdown` コンポーネントを再利用する前提でデザインする（コードブロック・箇条書き・見出し・インライン数式が出る）
- **ストリーミング**: トークンが少しずつ届く（タイプライター的に本文が伸びる）。既存の要約ストリーミング表示と同じ体験

## 3. コンポーネント別仕様

### 3-1. SessionList（左サイドバー）

過去セッションの一覧。**更新日時の降順**。

各行に表示するデータ:
- **タイトル**（LLM 自動生成。1〜2行で省略）
- **スコープ表示**: `all`（ライブラリ全体）/ `N papers`（特定文献 N 件）
- **相対更新日時**（例: `2h ago` / `yesterday`）
- 現在開いているセッションは選択状態（アクセント）

操作:
- 行クリックで開く
- 行のホバー/メニューから **タイトル編集** / **アーカイブ（ソフト削除）**
- 上部に `⊕ New chat` ボタン

空状態: セッションが1件もないとき「最初の Chat を始めよう」的なプレースホルダ + New chat CTA。

### 3-2. セッションヘッダ（右ペイン上部）

- **タイトル**（インライン編集可: クリックで編集モード）
- **ScopePicker**（後述）への入口（`scope: all` / `scope: 3 papers ▾`）
- 使用中の **provider / model** の控えめな表示（例: `Claude · sonnet`）

### 3-3. MessageList（会話本体）

メッセージは縦に積む。role により見た目を変える。

- **user**: 右寄せ or アクセント縁取りのバブル。プレーン寄りのテキスト
- **assistant**: 左寄せ。**Markdown + 数式**でリッチに描画。本文中/末尾に **ToolCallCard** が挟まる
- **tool**（ツール実行結果）: 単独では出さず、対応する ToolCallCard の中に畳む

スクロール: 新規トークン到着で**最下部に追従**（ユーザーが上にスクロール中は追従しない）。

### 3-4. ToolCallCard（最重要・要状態設計）

LLM のツール呼び出しを**全展開で可視化**する折りたたみカード。検索・DB書き換え・MCP の3系統すべてこのカードで表現する。設計すべき状態:

| 状態 | 見た目 |
|---|---|
| **提案・承認待ち**（`needs_approval=true`） | カードが**強調**（注意色）。引数プレビュー + **[許可] [拒否]** ボタン。承認するまで会話は進まない |
| **実行中** | スピナー + 「実行中…」 |
| **実行済み（折りたたみ）** | 1行サマリ（例: `🔍 fulltext_search("quantum walk") — 5 hits`）。クリックで展開 |
| **実行済み（展開）** | 引数（JSON）+ 結果プレビュー（検索ヒットのスニペット一覧 / 書き換え後の差分 / MCP のレスポンス） |
| **拒否済み** | 「拒否しました」の控えめな表示。会話は拒否を踏まえて続行 |

ツール種別でアイコン/色を分けると分かりやすい:
- 🔍 **読み取り系**（`fulltext_search` / `get_entry` / `list_*`）— 常に自動承認、地味でよい
- ✏️ **書き換え系**（`add_tag` / `update_notes` / `attach_ocr_text` / `add_to_collection`）— デフォルト自動だが「DBを変えた」ことが分かる色
- ⚠️ **要承認の書き換え**（`create_entry` / `update_entry`）— 承認待ち強調
- 🗑️ **削除**（`delete_*`）— 常に承認必須、最も強い警告色
- 🔌 **MCP**（`mcp_<server>_<tool>`）— 外部サーバー呼び出しと分かるバッジ。write 系は常に承認

承認ポリシーは「読み取り→自動 / 低リスク書き換え→自動 / `create`・`update`→都度承認 / `delete`・MCP write→常時承認」。**UI 上は『承認待ちかどうか』だけ意識すればよい**（判定はバックエンド）。

### 3-5. Composer（入力欄）

- 複数行対応のテキスト入力 + 送信ボタン
- **送信中（ストリーミング中）は送信ボタンが『■ 中断』に変わる** — 押すと進行中の応答を止める（部分応答は保存され、後から続けられる）
- 入力中の補助（任意）: scope の現在値を小さく示す

### 3-6. ScopePicker

セッションのコンテキスト範囲を切り替える。

- **all**: ライブラリ全体を検索対象にする
- **entries**: 指定した文献 N 件だけを対象にする → 文献の追加/削除ができるリスト（既存のエントリ選択 UI のトーンを踏襲）
- ヘッダのチップ or ポップオーバーで提供

### 3-7. NewSessionDialog（新規セッション作成）

- **プロバイダ / モデル**選択（OpenAI / Anthropic）
- **初期スコープ**: 全体 or 文献を選んで開始
- タイトルは空でよい（最初のターン後に自動生成される）

### 3-8. 設定（既存 Settings モーダルにタブ追加）

Chat 画面ではなく設定モーダル側だが、関連 UI として:
- **McpServerSettings**: 外部 MCP サーバーの追加・削除（`command` / `args` / `env`。Claude Desktop の `mcpServers` JSON 互換）。起動状態・取得済みツール一覧の表示
- **ChatToolWhitelist**: 自動承認するツールの上書き設定（`delete_*` と MCP write 系はトグル不可＝常に承認）

## 4. 主要ユーザーフロー（検証シナリオ由来）

設計の妥当性は以下のフローが破綻なく成立するかで判断する。

1. **複数文献で開始**: ライブラリで3件選択 →「これらで Chat 開始」→ セッション作成 →「共通点は？」→ assistant が `fulltext_search` を複数回呼び（ToolCallCard が並ぶ）、各文献のスニペットを引用しつつ回答 → カードを畳める
2. **全体検索**: 「DB全体」スコープで新規 →「量子ウォークについて」→ ライブラリ横断検索 → 関連文献を引用
3. **中断**: 長い応答の途中で『中断』→ 部分応答が残る、後で続行
4. **自動承認の書き換え**: 「entry #42 にタグ `quantum-walk` を追加して」→ ✏️カードが自動実行 → 結果表示
5. **要承認の書き換え**: 「entry #42 のタイトルを修正して」→ ⚠️カードが承認待ち → [拒否] で変更なし / [許可] で反映
6. **OCR 提案**: 「entry #99 を検索したい」→ 空ヒット → assistant が `ocr_pdf` を提案 → 承認 → 進捗 → 再検索で回答
7. **MCP**: 設定で Obsidian MCP を追加 → ツール一覧に `mcp_obsidian_*` → 「この発見を Obsidian の今日のノートに記録して」→ 🔌カードが承認待ち → 承認 → 反映
8. **永続化**: アプリ再起動 → サイドバーに過去セッションが残る → クリックで再開

## 5. 画面が描画するデータ（型）

UI はバックエンドから以下の形のデータを受け取る。空状態・省略・プレビュー文言の設計に使うこと。

```ts
type ChatRole = "user" | "assistant" | "tool";
type ScopeMode = "all" | "entries";

type ChatSession = {
  id: number;
  title: string;          // 自動生成、編集可。長文あり得る → 省略表示
  provider: string;       // "openai" | "anthropic"
  model: string;          // 例 "sonnet" / "gpt-4o-mini"
  scope_mode: ScopeMode;
  entry_count: number;    // scope_mode='entries' のとき N papers 表示
  updated_at: string;     // 相対表示に変換
};

type ToolCallSpec = {
  call_id: string;
  tool_name: string;      // "fulltext_search" / "add_tag" / "mcp_obsidian_append_note" 等
  arguments: Record<string, unknown>;  // 展開時に JSON 表示
};

type ChatMessage = {
  id: number;
  role: ChatRole;
  content: string;        // assistant は Markdown+数式
  tool_calls?: ToolCallSpec[];  // assistant に付随
  tool_call_id?: string;        // role='tool' の結果
};

// ストリーミングイベント（カードの状態遷移はこれで駆動される）
type ChatStreamEvent =
  | { kind: "session_started"; session_id: number }
  | { kind: "delta"; text: string }                       // 本文が伸びる
  | { kind: "tool_call_proposed"; call_id; tool_name; args_preview: string; needs_approval: boolean }  // カード出現
  | { kind: "tool_call_executed"; call_id; result_summary: string }  // カード→実行済み
  | { kind: "message_persisted"; message_id; role }
  | { kind: "done" }
  | { kind: "error"; message: string };                   // エラー表示
```

## 6. 入口（Chat 画面への遷移）

Chat 画面はこれらの経路から開く。各入口のアフォーダンスも設計対象:
- ライブラリ画面のグローバルナビ / サイドバー
- コマンドパレット（⌘K）: 「新規 Chat」「セッション一覧」
- ライブラリで複数選択中の一括操作パネル:「選択した N 件で Chat 開始」
- 詳細ビュー:「この文献で Chat」ボタン

## 7. デザインで判断してほしい論点

- ToolCallCard の**情報密度**（自動承認の読み取り系は会話を埋めすぎないトーンに、要承認は確実に目を引くトーンに）
- **承認待ちで会話が止まる**状態の見せ方（次のユーザー入力をブロックすべき）
- ストリーミング中 / 中断 / エラー の**状態遷移の視覚的連続性**
- light/dark × 4アクセント × 3密度の組み合わせで破綻しないこと
- 日本語・英語でラベル幅が変わっても崩れないこと
